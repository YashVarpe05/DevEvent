# Plan 4.1: Database Schema & Core Registration APIs

## Objective
Establish the `Registration` database model and implement the core attendee-facing API endpoints for registering and canceling event tickets.

## Context
- `database/event.model.ts`
- `database/user.model.ts`
- `lib/validations/`

## Tasks

<task type="auto">
  <name>Create Registration Model</name>
  <files>
    - database/registration.model.ts
  </files>
  <action>
    - Create the `Registration` Mongoose schema using the fields defined in Phase 4 prompt.
    - Fields: `eventId` (ObjectId), `attendeeUserId` (ObjectId), `attendeeEmail`, `attendeeName`, `status` (confirmed, cancelled_by_user, cancelled_by_organizer), `bookingType` (free), `quantity` (default 1), `ticketCode` (String, unique), `qrPayload` (String), `checkedInAt` (Date), `checkedInBy` (ObjectId), `source`, `metadata`.
    - Indexes:
      - Compound unique index: `{ eventId: 1, attendeeUserId: 1 }` to enforce one registration per user.
      - Unique index on `ticketCode`.
      - Index on `{ eventId: 1, createdAt: 1 }`.
  </action>
  <verify>Check `registration.model.ts` exists and compiles.</verify>
  <done>Mongoose model is strictly typed and exports a robust Schema.</done>
</task>

<task type="auto">
  <name>Implement Registration Validation Schemas</name>
  <files>
    - lib/validations/registration.ts
  </files>
  <action>
    - Create Zod schemas for registration.
    - `registerEventSchema`: body validation (even if empty for free tickets, good for future extensibility).
    - Create an idempotency or concurrency helper if necessary, though basic checks suffice for now.
  </action>
  <verify>Check schema existence.</verify>
  <done>Zod schemas are available for route validation.</done>
</task>

<task type="auto">
  <name>Implement Attendee Registration APIs</name>
  <files>
    - app/api/events/[id]/register/route.ts
    - app/api/registrations/me/route.ts
    - app/api/registrations/[id]/cancel/route.ts
  </files>
  <action>
    - **POST `/api/events/[id]/register`**: Require auth. Check event exists, is published, and within registration window. Check capacity (`confirmedCount < capacity`). Create `Registration` document with a unique `ticketCode` and placeholder `qrPayload`.
    - **GET `/api/registrations/me`**: Return all registrations for the logged-in user, populate event details.
    - **POST `/api/registrations/[id]/cancel`**: Allow attendee to cancel their own registration, updating status to `cancelled_by_user`.
  </action>
  <verify>Routes exist and handle errors gracefully returning standard HTTP statuses.</verify>
  <done>API endpoints return expected schema and validate user session.</done>
</task>

## Success Criteria
- [ ] Database schema is defined with correct indexes.
- [ ] Registration endpoint enforces capacity and duplicate user checks.
- [ ] Attendee can retrieve their registrations via API.
