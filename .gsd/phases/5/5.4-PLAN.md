# Plan 5.4: Webhooks & Inventory Fulfillment

## Objective
Implement resilient Stripe webhooks to handle payment completion, failure, and inventory decrementing.

## Context
- `app/api/webhooks/stripe/route.ts`
- `database/order.model.ts`
- `database/registration.model.ts`

## Tasks

<task type="auto">
  <name>Build Secure Stripe Webhook API</name>
  <files>
    - app/api/webhooks/stripe/route.ts
  </files>
  <action>
    - Use `stripe.webhooks.constructEvent` with `STRIPE_WEBHOOK_SECRET` for signature verification.
    - Implement a basic idempotency check (to handle duplicate Stripe events).
    - Map events: `checkout.session.completed`, `payment_intent.succeeded`, `payment_intent.payment_failed`.
  </action>
  <verify>Stripe CLI `listen` tool confirms signature verification passes.</verify>
  <done>System responds correctly to real-time Stripe events.</done>
</task>

<task type="auto">
  <name>Implement Fulfillment State Transitions</name>
  <files>
    - lib/fulfillment.ts
    - app/api/webhooks/stripe/route.ts
  </files>
  <action>
    - Transition Order to `paid`.
    - Atomically increment `quantitySold` in `TicketType`.
    - Create `Registration` document(s) and issue `ticketCode`.
    - Create `PaymentTransaction` (ledger) entry.
  </action>
  <verify>Stripe CLI `trigger checkout.session.completed` results in a new confirmed registration.</verify>
  <done>User is booked only after successful payment is confirmed by Stripe.</done>
</task>

<task type="auto">
  <name>Configure Refund Webhooks</name>
  <files>
    - app/api/webhooks/stripe/route.ts
  </files>
  <action>
    - Map `charge.refunded` event.
    - Transition Order to `refunded`.
    - Increment remaining capacity (decrement `quantitySold`).
    - Mark registrations as `cancelled_by_organizer` (due to refund).
  </action>
  <verify>Stripe CLI `trigger charge.refunded` updates order status correctly.</verify>
  <done>Refunds are automatically reflected in the inventory.</done>
</task>

## Success Criteria
- [ ] Webhook signature is strictly verified.
- [ ] Multiple payments do not result in over-booking.
- [ ] Order status accurately reflects the final payment state.
